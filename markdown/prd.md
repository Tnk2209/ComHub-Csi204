# เอกสารข้อกำหนดความต้องการผลิตภัณฑ์ (Product Requirement Document - PRD)

เอกสารข้อกำหนดความต้องการผลิตภัณฑ์นี้สังเคราะห์ขึ้นเพื่อใช้เป็นโครงสร้างข้อกำหนดการวิเคราะห์ความต้องการและการออกแบบระบบ **ComHub** ตามแผนงาน SDLC วิชา **csi 204**

---

## 1. คำแถลงปัญหา (Problem Statement)
* **มุมมองของลูกค้า (Customer):** ลูกค้าส่วนใหญ่ที่ต้องการซื้อคอมพิวเตอร์ประกอบเอง มักจะไม่ทราบความเข้ากันได้ทางเทคนิคของอุปกรณ์ฮาร์ดแวร์ไอทีแต่ละประเภท (เช่น Socket ขาของ CPU และเมนบอร์ดไม่ตรงกัน หรือกำลังวัตต์ของ Power Supply ไม่เพียงพอ) ส่งผลให้ซื้อของผิดพลาดและใช้งานไม่ได้จริง
* **มุมมองของพนักงาน (Staff & Management):** พนักงานประกอบเครื่องไม่มีระบบส่วนกลางในการจัดคิวงานประกอบ ตลอดจนการทำระบบทดสอบความร้อนก่อนจัดส่งสินค้า (Burn-in Test UAT) ขาดความเป็นระบบ ไม่มีบันทึกอุณหภูมิที่ชัดเจน และผู้จัดการร้านไม่สามารถวัดสถิติยอดขายหรือสร้างโปรโมชั่นคอมพิวเตอร์เซ็ตแนะนำได้สะดวก

---

## 2. แนวทางแก้ไข (Solution)
สร้างแพลตฟอร์ม **ComHub** ซึ่งเป็นระบบร้านค้าออนไลน์ (E-Commerce) ควบคู่กับเครื่องมือช่วยจัดสเปคคอมพิวเตอร์อัจฉริยะ (Advanced PC Builder) ที่ประมวลตรรกะความเข้ากันได้ของสเปคและคำนวณกำลังไฟแบบเรียลไทม์ พร้อมระบบควบคุมคิวงานสำหรับช่างประกอบหลังบ้าน หน้า UAT บันทึกอุณหภูมิการตรวจความร้อน และแกลเลอรี่คอมมูนิตี้สำหรับแชร์-คัดลอกสเปคคอมพิวเตอร์อ้างอิง

---

## 3. เรื่องราวของผู้ใช้งาน (User Stories)

### 3.1 ฝั่งลูกค้า (Customer)
1. **As a** ลูกค้าทั่วไป, **I want to** ค้นหาและกรองชิ้นส่วนสินค้าไอทีแยกหมวดหมู่, **so that** ฉันสามารถเลือกซื้ออุปกรณ์เดี่ยวได้ตามต้องการ
2. **As a** ลูกค้าทั่วไป, **I want to** เข้าใช้งานตัวช่วยจัดสเปคคอมพิวเตอร์ (PC Builder) ทีละส่วน, **so that** ฉันสามารถแต่งสเปคเครื่องในฝันได้ทีละชิ้นส่วน
3. **As a** ลูกค้าทั่วไป, **I want to** ให้ระบบบล็อกไม่ให้เลือกเมนบอร์ดและแรมที่ชนิดต่างกัน (เช่น DDR4 กับ DDR5), **so that** ฉันจะไม่สั่งซื้อของที่ประกอบกันไม่ได้
4. **As a** ลูกค้าทั่วไป, **I want to** ให้ระบบแสดงข้อความเตือนเมื่อเลือกการ์ดจอที่ยาวเกินขนาดขีดจำกัดของเคส, **so that** ฉันสามารถมั่นใจได้ว่าอุปกรณ์จะใส่ลงเคสคอมพิวเตอร์ได้พอดี
5. **As a** ลูกค้าทั่วไป, **I want to** ให้ระบบคำนวณรวมกำลังวัตต์ TDP ของสินค้าที่กำลังจัด และแนะนำเฉพาะ Power Supply ที่จ่ายไฟพอเพียง (+ เผื่อ 20%), **so that** คอมพิวเตอร์ของฉันจะไม่มีปัญหาไฟวูบขณะทำงานหนัก
6. **As a** ลูกค้าทั่วไป, **I want to** เลือกตัวเลือกสั่งซื้อแบบ "ให้ทางร้านประกอบเครื่องให้", **so that** ฉันจะได้รับคอมพิวเตอร์จัดสำเร็จรูปที่ผ่านการทดสอบมาแล้ว
7. **As a** ลูกค้าทั่วไป, **I want to** ใส่รหัสคูปองส่วนลดที่หน้าออเดอร์, **so that** ฉันจะได้รับส่วนลดราคาสุดพิเศษตามโปรโมชั่น
8. **As a** ลูกค้าทั่วไป, **I want to** แนบภาพสลิปโอนเงินชำระเงินเข้ามาในออเดอร์, **so that** พนักงานสามารถนำไปตรวจสอบการจ่ายเงินได้
9. **As a** ลูกค้าทั่วไป, **I want to** ดูแถบความคืบหน้าติดตามการประกอบ 4 ขั้นตอน, **so that** ฉันสามารถรู้ความเคลื่อนไหวออเดอร์ได้อย่างต่อเนื่อง
10. **As a** ลูกค้าทั่วไป, **I want to** เปิดแถบดูประวัติล็อกเวลาสลับสถานะ และผลอุณหภูมิ CPU/GPU ที่ช่างบันทึก UAT ไว้, **so that** ฉันจะมั่นใจว่าเครื่องประกอบเสร็จผ่านการเทสความร้อนมาแล้วจริง
11. **As a** ลูกค้าทั่วไป, **I want to** เพิ่มสินค้าที่หมดคลังเข้าในรายการโปรด (Wishlist) และกดเปิดระบบสัญญาณเตือนเมื่อสต็อกเข้ามาใหม่, **so that** ฉันจะได้รับป้ายแจ้งเตือนทันทีเมื่อของเข้ามาพร้อมสั่งซื้อ
12. **As a** ลูกค้าทั่วไป, **I want to** เขียนรีวิว ให้ดาว (1-5) และลงรูปภาพประกอบรีวิวสินค้า, **so that** ฉันสามารถแชร์ประสบการณ์ใช้งานให้ลูกค้าท่านอื่นทราบ
13. **As a** ลูกค้าทั่วไป, **I want to** แชร์สเปคคอมพิวเตอร์ประกอบเสร็จของตนลงสู่แกลเลอรี่คอมมูนิตี้สาธารณะ, **so that** ฉันสามารถโชว์ผลงานคอมพิวเตอร์ให้เพื่อนเข้ามาชมและกดถูกใจได้
14. **As a** ลูกค้าทั่วไป, **I want to** กดคัดลอกสเปคคอมพิวเตอร์จากแกลเลอรี่ของคนอื่นเข้าสู่ PC Builder ของฉัน, **so that** ฉันสามารถสั่งซื้อสินค้าทั้งหมดตามเครื่องนั้นได้ทันทีโดยไม่ต้องจัดเองใหม่
15. **As a** ลูกค้าทั่วไป, **I want to** เห็นป้ายกำกับสินค้าที่เลิกจำหน่าย (Disable) บนโพสต์แกลเลอรี่, **so that** ฉันจะทราบได้ว่าชิ้นส่วนบางชิ้นไม่มีจำหน่ายในคลังปัจจุบันแล้ว

### 3.2 ฝั่งพนักงานจัดประกอบเครื่อง (Staff)
16. **As a** พนักงานจัดประกอบ, **I want to** ดูรายการออเดอร์ที่ลูกค้าขอรับบริการประกอบและชำระเงินเรียบร้อยแล้ว, **so that** ฉันสามารถจัดทำและหยิบสินค้ามาประกอบตามคิวงานได้
17. **As a** พนักงานจัดประกอบ, **I want to** บันทึกค่าอุณหภูมิ CPU และ GPU พร้อมประเมินผลผ่าน/ไม่ผ่านหลังทำการ Burn-in Test, **so that** ฉันสามารถบันทึกรายงานการควบคุมคุณภาพของออเดอร์ได้
18. **As a** พนักงานจัดประกอบ, **I want to** ป้อนข้อมูลเลขที่จัดส่งพัสดุ (Tracking Number), **so that** ลูกค้าสามารถนำไปใช้ติดตามพัสดุกับบริษัทขนส่งภายนอกได้

### 3.3 ฝั่งผู้จัดการร้าน (Manager)
19. **As a** ผู้จัดการร้าน, **I want to** สร้าง จัดการแก้ไข และลบชุดสเปคคอมพิวเตอร์จัดแนะนำของทางร้าน (Pre-built Templates), **so that** ฉันสามารถโปรโมตเซ็ตคอมพิวเตอร์ตามระดับราคาต่างๆ ให้ลูกค้าสั่งซื้อได้ง่ายขึ้น
20. **As a** ผู้จัดการร้าน, **I want to** ตรวจสอบสกรีนภาพถ่ายรีวิวและรูปแกลเลอรี่ประกอบคอมพิวเตอร์ของผู้ใช้, **so that** ฉันสามารถสกัดกั้นภาพที่ไม่สุภาพหรือผิดกฎไม่ให้แสดงสู่สาธารณะได้
21. **As a** ผู้จัดการร้าน, **I want to** ดูแดชบอร์ดสรุปผลยอดขาย สินค้าขายดี และรายการแจ้งเตือนของในคลังใกล้หมด, **so that** ฉันสามารถวางแผนการสั่งซื้อและการตลาดได้อย่างถูกต้อง

### 3.4 ฝั่งผู้ดูแลระบบ (Admin)
22. **As a** ผู้ดูแลระบบ, **I want to** เพิ่ม ลบ แก้ไข และกดสลับสถานะเปิดใช้งาน (is_active) ของสินค้าไอที, **so that** ฉันสามารถควบคุมข้อมูลคลังสินค้าของร้านได้ทันที
23. **As a** ผู้ดูแลระบบ, **I want to** จัดการสิทธิ์การเข้าถึงข้อมูลและสร้างบัญชีเพิ่มให้ Staff และ Manager, **so that** ฉันสามารถกระจายงานควบคุมระบบได้ตามหน้าที่

## 4. การตัดสินใจเชิงระบบหลังบ้านและการติดตั้ง (Implementation Decisions)

### 4.1 สถาปัตยกรรมข้อมูลหลัก (Core Schemas)
* **ตารางสินค้าและสเปคแบบยืดหยุ่น (Products specifications JSONB):** ยุบคุณสมบัติฮาร์ดแวร์ไอทีที่หลากหลาย (Socket, Form Factor, Max GPU Length, TDP, output wattage) เข้าไปในคอลัมน์ JSONB คอลัมน์เดียว ช่วยลดความซับซ้อนของการปรับปรุง Table ในอนาคต
* **การลบสินค้า (Soft Delete via `is_active`):** ใช้ฟิลด์ Boolean `is_active` ควบคุมการแสดงผลในหน้าแค็ตตาล็อกช้อปปิ้งออนไลน์ สินค้าเก่าที่ปิดตัวจะยังถูกดึงมาประกอบประวัติคำสั่งซื้อและโพสต์ในแกลเลอรี่ได้ปกติ โดยมีแท็ก Badges ป้ายคำกำกับสีส้มแสดงผลในหน้าแกลเลอรี่ว่า `[เลิกจำหน่าย]` เพื่อแจ้งเตือนผู้คัดลอกสเปคคอมพิวเตอร์
* **ตารางเชื่อม Many-to-Many (`TemplateItems`):** สร้างตารางช่วยจับคู่เชื่อมสัมพันธ์ของ Prebuilt Template กับอุปกรณ์แบบไม่จำกัดจำนวนชิ้น
* **การแยกแยะสถานะซื้อและการประกอบ (`Orders`):** คอนเซ็ปต์ตาราง Orders แยกเป็น `payment_status` (Pending/Approved/Rejected) และ `order_status` (Pending/Assembling/Testing/Shipped) การก้าวเข้าสู่กระบวนการประกอบจะถูกบล็อกทางเงื่อนไขจนกว่าสถานะชำระเงินจะถูกเซ็ตเป็น `Approved`
* **ตารางบันทึกการทำงานประวัติและการแชร์โพสต์แกลเลอรี่ (`OrderLogs` & `GalleryPosts`):** 
  - ตาราง `OrderLogs` จะเก็บบันทึกประวัติช่วงวันเวลาและตัวพนักงานผู้สับเปลี่ยนสถานะย้อนหลังเป็นลำดับไทม์ไลน์
  - ตาราง `GalleryPosts` แยกเฉพาะเพื่อผูกสัมพันธ์แบบแชร์สเปคออเดอร์จาก `order_id` และผ่านการตรวจสอบความเหมาะสมรูปภาพจาก Manager ก่อนโชว์ขึ้นแสดงผลเว็บบอร์ด

### 4.2 ระบบความเข้ากันได้แรม (RAM Compatibility Logic)
* ในตัวสินค้าหมวด CPU ฟิลด์สเปคจะเก็บค่าความเข้ากันได้ชนิดแรมแบบ Array สเปค เช่น `supported_ram: ["DDR4", "DDR5"]`
* บอร์ดและแรมเดี่ยวจะระบุชนิดแรมเพียงค่าเดียว เช่น `ram_type: "DDR4"` หรือ `ram_type: "DDR5"`
* ตรรกะตรวจเช็คความเข้ากันได้จะทำการจับคู่เปรียบเทียบ `Motherboard.specifications.ram_type == RAM.specifications.ram_type` และตรวจสอบความยินยอมของตัวประมวลผลกลางว่า `CPU.specifications.supported_ram` บรรจุค่าแรมประเภทดังกล่าวอยู่หรือไม่

## 5. ขอบเขตการทดสอบระบบ (Testing Decisions)
* **ตรรกะการตรวจสอบความเข้ากันได้ (PC Builder logic validation):** จะถูกทดสอบผ่านทาง Unit Testing โดยเขียนชุดการทดสอบป้อนค่าโมเดลสเปคที่คาดว่าจะผ่าน และกลุ่มโมเดลที่จะต้องบล็อกการเลือกซื้อ (เช่น บอร์ด LGA1700 กับ CPU AM5 หรือ แรมต่างตระกูล)
* **ตรรกะการคำนวณ TDP ไฟวัตต์สะสม (TDP Calculations):** เขียนชุดการทดสอบการรวมค่าผลลัพธ์ TDP และการเปรียบเทียบผลคัดเลือก Power Supply ที่มีกำลังขับมากกว่าผลคูณค่าเผื่อความปลอดภัย 1.2 เท่า
* **การสกัดกั้นสิทธิ์ผู้ใช้งาน (RBAC security test):** ทดสอบผ่านกระบวนการจำลองยิง Request ไปที่ API (Integration Test) เพื่อพิสูจน์ว่าผู้ใช้บทบาททั่วไปไม่สามารถแอบอ้างสิทธิ์พนักงานเพื่อเปลี่ยนสเตตัสการประกอบหรือสลิปชำระเงินของคนอื่นได้

---

## 6. ข้อกำหนดอยู่นอกขอบเขตของโปรเจ็ค (Out of Scope)
* **การต่อพ่วงเกตเวย์รับจ่ายเงินจริง (Real Payment Gateway API Integration):** จะอยู่นอกขอบเขตของโครงการ โดยระบบจะใช้วิธีการอัปโหลดไฟล์ภาพสลิปจำลองขึ้นคลาวด์เพื่อให้เจ้าหน้าที่ตรวจสอบด้วยการมองภาพแทน
* **ระบบยิงอีเมลหรือข้อความการขนส่งจริง (Real Logistics & Notifications Service):** ระบบแจ้งเตือนสต็อกของโปรดและประวัติจะใช้การแจ้งเตือนแบบป็อปอัปหน้าจอ (UI notification toasts / notifications box) ภายในเว็บไซต์โดยตรง

---

## 7. สิ่งที่ระบบยังขาด (Missing Features & Gap Analysis)
จากการวิเคราะห์ระบบ พบว่าสเปคความเข้ากันได้และการดำเนินการสั่งซื้อมีช่องโหว่ทางธุรกิจดังต่อไปนี้ที่ต้องระบุและปิดช่องว่างในคู่มือการออกแบบ:
1. **คูปองหมดอายุหรือจำกัดสิทธิ์ (Coupon Validation Guard):** ในตารางออเดอร์มีฟิลด์เก็บชื่อคูปองและส่วนลด แต่ตารางสิทธิ์คูปองยังไม่มีระบบตั้งระยะเวลาเปิดใช้งานหรือเกณฑ์จำกัดโควต้าการใช้คูปองต่อสมาชิก
2. **การคืนค่าสินค้าเข้าคลังกรณีสลิปโอนเงินถูกปฏิเสธ (Stock Rollback on Slip Rejection):** เมื่อลูกค้ากดสั่งซื้อ สินค้าในสต็อกจะถูกหักออกทันที แต่ในกรณีที่พนักงานตรวจสอบแล้วสลิปเป็นข้อมูลเท็จและเปลี่ยนสถานะเป็น Rejected ระบบต้องมีฟังก์ชันคืนจำนวนสินค้ากลับเข้าสต็อก
3. **การป้องกันการคัดลอกสเปคคอมแกลเลอรี่ที่มีสินค้าปิดขาย (Active Gallery Filter Guard):** ในฟังก์ชันดึงสเปคคอมของเพื่อนคนอื่นไปใช้งานใน PC Builder หากสเปคนั้นมีชิ้นส่วนใดชิ้นส่วนหนึ่งที่มีสถานะ `is_active = false` (เลิกจำหน่าย) ระบบจะต้องดึงมาเฉพาะรายการสินค้าชิ้นอื่นๆ ที่ยังขายได้ และแจ้งเตือนให้ลูกค้าเลือกชิ้นส่วนทดแทนในหมวดหมู่ที่ขาดหายไปนั้น
