# เอกสารข้อกำหนดความต้องการ (Product Requirement Document - PRD) - MVP Version

เอกสารข้อกำหนดความต้องการผลิตภัณฑ์นี้สังเคราะห์ขึ้นเพื่อใช้เป็นโครงสร้างข้อกำหนดการวิเคราะห์ความต้องการและการออกแบบระบบ **ComHub** ตามแผนงาน SDLC วิชา **csi 204**

> **📝 หมายเหตุ MVP Scope:** เอกสารนี้ได้ปรับลดเพื่อให้เหมาะสมกับกรอบเวลา โดยเน้น 2 บทบาทหลัก (Customer และ Admin) และตัดฟีเจอร์ที่ไม่จำเป็นออก

---

## 1. ปัญหา (Problem Statement)

* **มุมมองของลูกค้า (Customer):** ลูกค้าส่วนใหญ่ที่ต้องการซื้อคอมพิวเตอร์ประกอบเอง มักจะไม่ทราบความเข้ากันได้ทางเทคนิคของอุปกรณ์ฮาร์ดแวร์ไอทีแต่ละประเภท (เช่น Socket ขาของ CPU และเมนบอร์ดไม่ตรงกัน หรือกำลังวัตต์ของ Power Supply ไม่เพียงพอ) ส่งผลให้ซื้อของผิดพลาดและใช้งานไม่ได้จริง
* **มุมมองของ Admin:** ต้องการระบบที่สามารถจัดการสินค้า ตรวจสอบการชำระเงิน อัปเดตสถานะออเดอร์ และดูภาพรวมธุรกิจได้อย่างมีประสิทธิภาพ

---

## 2. แนวทางแก้ไข (Solution)

สร้างแพลตฟอร์ม **ComHub MVP** ซึ่งเป็นระบบร้านค้าออนไลน์ (E-Commerce) ควบคู่กับเครื่องมือช่วยจัดสเปคคอมพิวเตอร์อัจฉริยะ (Advanced PC Builder) ที่ประมวลตรรกะความเข้ากันได้ของสเปคและคำนวณกำลังไฟแบบเรียลไทม์ พร้อมระบบจัดการหลังบ้านสำหรับ Admin

---

## 3. เรื่องราวของผู้ใช้งาน (User Stories)

### 3.1 ฝั่งลูกค้า (Customer)

1. **As a** ลูกค้าทั่วไป, **I want to** ค้นหาและกรองชิ้นส่วนสินค้าไอทีแยกหมวดหมู่, **so that** ฉันสามารถเลือกซื้ออุปกรณ์เดี่ยวได้ตามต้องการ
2. **As a** ลูกค้าทั่วไป, **I want to** เข้าใช้งานตัวช่วยจัดสเปคคอมพิวเตอร์ (PC Builder) ทีละส่วน, **so that** ฉันสามารถแต่งสเปคเครื่องในฝันได้ทีละชิ้นส่วน
3. **As a** ลูกค้าทั่วไป, **I want to** ให้ระบบบล็อกไม่ให้เลือกเมนบอร์ดและแรมที่ชนิดต่างกัน (เช่น DDR4 กับ DDR5), **so that** ฉันจะไม่สั่งซื้อของที่ประกอบกันไม่ได้
4. **As a** ลูกค้าทั่วไป, **I want to** ให้ระบบแสดงข้อความเตือนเมื่อเลือกการ์ดจอที่ยาวเกินขนาดขีดจำกัดของเคส, **so that** ฉันสามารถมั่นใจได้ว่าอุปกรณ์จะใส่ลงเคสคอมพิวเตอร์ได้พอดี
5. **As a** ลูกค้าทั่วไป, **I want to** ให้ระบบคำนวณรวมกำลังวัตต์ TDP ของสินค้าที่กำลังจัด และแนะนำเฉพาะ Power Supply ที่จ่ายไฟพอเพียง (+ เผื่อ 20%), **so that** คอมพิวเตอร์ของฉันจะไม่มีปัญหาไฟวูบขณะทำงานหนัก
6. **As a** ลูกค้าทั่วไป, **I want to** เปรียบเทียบสเปคสินค้า 2-3 ชิ้นแบบเคียงข้าง, **so that** ฉันสามารถตัดสินใจเลือกซื้อได้ง่ายขึ้น
7. **As a** ลูกค้าทั่วไป, **I want to** แนบภาพสลิปโอนเงินชำระเงินเข้ามาในออเดอร์ (Mockup), **so that** Admin สามารถนำไปตรวจสอบการจ่ายเงินได้
8. **As a** ลูกค้าทั่วไป, **I want to** ดูแถบความคืบหน้าติดตามออเดอร์ 5 ขั้นตอน (Pending Payment → Paid → Processing → Shipped → Delivered), **so that** ฉันสามารถรู้ความเคลื่อนไหวออเดอร์ได้อย่างต่อเนื่อง
9. **As a** ลูกค้าทั่วไป, **I want to** เพิ่มสินค้าเข้าในรายการโปรด (Wishlist), **so that** ฉันสามารถบันทึกสินค้าที่สนใจไว้สั่งซื้อภายหลัง (MVP: ไม่มี Stock Alert)
10. **As a** ลูกค้าทั่วไป, **I want to** เขียนรีวิว ให้ดาว (1-5) และข้อความ, **so that** ฉันสามารถแชร์ประสบการณ์ใช้งานให้ลูกค้าท่านอื่นทราบ

### 3.2 ฝั่งผู้ดูแลระบบ (Admin)

13. **As a** ผู้ดูแลระบบ, **I want to** เพิ่ม ลบ แก้ไข และกดสลับสถานะเปิดใช้งาน (is_active) ของสินค้าไอที, **so that** ฉันสามารถควบคุมข้อมูลคลังสินค้าของร้านได้ทันที
14. **As a** ผู้ดูแลระบบ, **I want to** อนุมัติหรือปฏิเสธสลิปโอนเงินของลูกค้า, **so that** ฉันสามารถตรวจสอบการชำระเงินก่อนดำเนินการจัดส่ง
15. **As a** ผู้ดูแลระบบ, **I want to** อัปเดตสถานะออเดอร์และกรอก Tracking Number, **so that** ลูกค้าสามารถติดตามพัสดุได้
16. **As a** ผู้ดูแลระบบ, **I want to** ดู Dashboard สรุปยอดขาย สินค้ายอดนิยม และสต็อกต่ำ, **so that** ฉันสามารถวางแผนการจัดการธุรกิจได้
17. **As a** ผู้ดูแลระบบ, **I want to** สร้างและจัดการบัญชี Admin เพิ่มเติม (สร้าง แก้ไขสิทธิ์ Customer/Admin ลบบัญชี), **so that** ฉันสามารถกระจายงานให้ผู้ดูแลระบบคนอื่นได้ (A-04)

### 3.3 ระบบรองรับ (System-wide)

18. **As a** ผู้ใช้ทั่วไป, **I want to** ให้ระบบบีบอัดรูปภาพสลิปที่ฝั่งเบราว์เซอร์เป็น **WebP คุณภาพ 80%** ก่อนแปลงเป็น Base64, **so that** ลดพื้นที่จัดเก็บและไม่ต้องพึ่ง Cloud Storage สำหรับ MVP (SYS-04)

## 4. การตัดสินใจเชิงระบบหลังบ้านและการติดตั้ง (Implementation Decisions)

### 4.1 สถาปัตยกรรมข้อมูลหลัก (Core Schemas)

* **ตารางสินค้าและสเปคแบบยืดหยุ่น (Products specifications JSONB):** ยุบคุณสมบัติฮาร์ดแวร์ไอทีที่หลากหลาย (Socket, Form Factor, Max GPU Length, TDP, output wattage) เข้าไปในคอลัมน์ JSONB คอลัมน์เดียว ช่วยลดความซับซ้อนของการปรับปรุง Table ในอนาคต
* **การลบสินค้า (Soft Delete via `is_active`):** ใช้ฟิลด์ Boolean `is_active` ควบคุมการแสดงผลในหน้าแค็ตตาล็อกช้อปปิ้งออนไลน์ สินค้าเก่าที่ปิดตัวจะยังถูกดึงมาประกอบประวัติคำสั่งซื้อและโพสต์ในแกลเลอรี่ได้ปกติ โดยมีแท็ก Badges ป้ายคำกำกับสีส้มแสดงผลในหน้าแกลเลอรี่ว่า `[เลิกจำหน่าย]` เพื่อแจ้งเตือนผู้คัดลอกสเปคคอมพิวเตอร์
* **การแยกแยะสถานะซื้อและการจัดส่ง (`Orders`):** คอนเซ็ปต์ตาราง Orders แยกเป็น `payment_status` (Pending/Approved/Rejected) และ `order_status` 5 ขั้นตาม MVP (`Pending Payment` → `Paid` → `Processing` → `Shipped` → `Delivered`) การเลื่อนสถานะออเดอร์ไป `Processing` จะถูกบล็อกทางเงื่อนไขจนกว่าสถานะชำระเงินจะถูกเซ็ตเป็น `Approved`
* **การจัดเก็บสลิปโอนเงิน (`payment_slip_mockup`):** ในตาราง `orders` มีคอลัมน์เดียวเก็บสลิปเป็น **Base64 encoded WebP** (Client-side Compression) เพื่อลดขนาดไฟล์และไม่พึ่งพา Cloud Storage สำหรับ MVP
* **ตารางบันทึกประวัติการเปลี่ยนสถานะ (`OrderLogs`):** เก็บบันทึกไทม์ไลน์วันเวลาที่ Admin สับเปลี่ยนสถานะ (Approve สลิป, Processing, Shipped, Delivered) พร้อมอ้างอิงผู้ปรับปรุงผ่าน `changed_by_user_id`
* **ตะกร้าสินค้าและการเปรียบเทียบ (Cart & Comparison):** ทั้ง Cart Items และ Comparison List จัดเก็บที่ฝั่ง Client (LocalStorage) เท่านั้น ไม่มีตารางใน DB (ตาม MVP scope)

### 4.2 ระบบความเข้ากันได้แรม (RAM Compatibility Logic)

* ในตัวสินค้าหมวด CPU ฟิลด์สเปคจะเก็บค่าความเข้ากันได้ชนิดแรมแบบ Array สเปค เช่น `supported_ram: ["DDR4", "DDR5"]`
* บอร์ดและแรมเดี่ยวจะระบุชนิดแรมเพียงค่าเดียว เช่น `ram_type: "DDR4"` หรือ `ram_type: "DDR5"`
* ตรรกะตรวจเช็คความเข้ากันได้จะทำการจับคู่เปรียบเทียบ `Motherboard.specifications.ram_type == RAM.specifications.ram_type` และตรวจสอบความยินยอมของตัวประมวลผลกลางว่า `CPU.specifications.supported_ram` บรรจุค่าแรมประเภทดังกล่าวอยู่หรือไม่

## 5. ขอบเขตการทดสอบระบบ (Testing Decisions)

* **ตรรกะการตรวจสอบความเข้ากันได้ (PC Builder logic validation):** จะถูกทดสอบผ่านทาง Unit Testing โดยเขียนชุดการทดสอบป้อนค่าโมเดลสเปคที่คาดว่าจะผ่าน และกลุ่มโมเดลที่จะต้องบล็อกการเลือกซื้อ (เช่น บอร์ด LGA1700 กับ CPU AM5 หรือ แรมต่างตระกูล)
* **ตรรกะการคำนวณ TDP ไฟวัตต์สะสม (TDP Calculations):** เขียนชุดการทดสอบการรวมค่าผลลัพธ์ TDP และการเปรียบเทียบผลคัดเลือก Power Supply ที่มีกำลังขับมากกว่าผลคูณค่าเผื่อความปลอดภัย 1.2 เท่า
* **การสกัดกั้นสิทธิ์ผู้ใช้งาน (RBAC security test):** ทดสอบผ่านกระบวนการจำลองยิง Request ไปที่ API (Integration Test) เพื่อพิสูจน์ว่าผู้ใช้บทบาท `Customer` ไม่สามารถแอบอ้างสิทธิ์ `Admin` เพื่อเปลี่ยนสถานะออเดอร์ อนุมัติสลิป หรือเข้าถึงฟังก์ชันหลังบ้านของลูกค้าคนอื่นได้
* **การบีบอัดรูปฝั่ง Client (WebP Compression Test):** ทดสอบว่ารูปภาพที่อัปโหลด (สลิป) ถูกบีบอัดผ่าน Canvas API เป็นไฟล์ WebP คุณภาพ 80% ก่อน encode เป็น Base64 และมีขนาดไม่เกินเกณฑ์ที่กำหนด

## 6. ข้อกำหนดอยู่นอกขอบเขตของโปรเจ็ค (Out of Scope)

* **การต่อพ่วงเกตเวย์รับจ่ายเงินจริง (Real Payment Gateway API Integration):** จะอยู่นอกขอบเขตของโครงการ โดยระบบจะใช้วิธีการอัปโหลดไฟล์ภาพสลิปจำลองขึ้นคลาวด์เพื่อให้เจ้าหน้าที่ตรวจสอบด้วยการมองภาพแทน
* **ระบบยิงอีเมลหรือข้อความการขนส่งจริง (Real Logistics & Notifications Service):** ระบบแจ้งเตือนสต็อกของโปรดและประวัติจะใช้การแจ้งเตือนแบบป็อปอัปหน้าจอ (UI notification toasts / notifications box) ภายในเว็บไซต์โดยตรง

---

## 7. สิ่งที่ระบบยังขาด (Missing Features & Gap Analysis)

จากการวิเคราะห์ระบบ พบว่าสเปคความเข้ากันได้และการดำเนินการสั่งซื้อมีช่องโหว่ทางธุรกิจดังต่อไปนี้ที่ต้องระบุและปิดช่องว่างในคู่มือการออกแบบ:

1. **การคืนค่าสินค้าเข้าคลังกรณีสลิปโอนเงินถูกปฏิเสธ (Stock Rollback on Slip Rejection):** เมื่อลูกค้ากดสั่งซื้อ สินค้าในสต็อกจะถูกหักออกทันที แต่ในกรณีที่ Admin ตรวจสอบแล้วสลิปเป็นข้อมูลเท็จและเปลี่ยนสถานะเป็น Rejected ระบบต้องมีฟังก์ชันคืนจำนวนสินค้ากลับเข้าสต็อก
2. **การจัดการการชนกันของสต็อกกรณีเปิดหลายแท็บ (Concurrent Cart Guard):** เนื่องจากตะกร้าสินค้าเก็บไว้ใน LocalStorage ฝั่ง Client หากผู้ใช้เปิดหลายแท็บพร้อมกันแล้วเช็คเอาต์ ระบบต้องตรวจสอบสต็อกล่าสุดที่ Backend ก่อนสร้างออเดอร์ (Overselling Prevention)
3. **การป้องกันการรีวิวก่อนได้รับสินค้า (Review Guard):** ระบบต้องอนุญาตให้เขียนรีวิวเฉพาะออเดอร์ที่มีสถานะ `Delivered` เท่านั้น (ผูกกับ `order_id` ในตาราง `reviews`)

---

## 8. ข้อพิจารณาที่ไม่ใช่ฟีเจอร์หลัก (Non-Functional Considerations — Future Version)

- **Email Verification (OTP):** การยืนยันตัวตนผ่าน OTP อีเมลหลังสมัครสมาชิก — วางแผนไว้สำหรับ version ถัดไป ยังไม่อยู่ใน scope การพัฒนาหรือการทดสอบ (UAT) ปัจจุบัน ต้องการ email service provider เพิ่มเติมใน backend infrastructure
